
# 📝 时间管理语音日志小程序 需求规格说明书 (SRS)

## 1. 引言 (Introduction)

### 1.1. 目的
本文档旨在详细描述“时间管理语音日志小程序”的功能、性能、设计约束和设计属性，作为软件开发、测试和最终验收的正式依据。

### 1.2. 范围
本软件是一个基于**微信/支付宝等平台的小程序**，核心功能是利用语音快速记录用户的时间使用情况，并提供自动化的日志生成、多维度回顾和可视化复盘总结功能。

### 1.3. 目标用户
* 时间管理需求强烈、追求效率的职场人士。
* 希望通过记录来可视化时间分配、进行深度复盘的用户。
* 追求操作便捷性、无法忍受复杂记录流程的用户。

### 1.4. 灵感来源
基于《时间黑洞》一书中的时间管理理念，实现对时间的精细化、便捷式记录与管理。

---

## 2. 总体描述 (Overall Description)

### 2.1. 产品愿景
通过最便捷的语音输入方式，帮助用户高效捕捉时间流逝的瞬间，将时间可视化、可量化，最终达到时间管理与效率提升的目的。

### 2.2. 产品架构（逻辑功能模块）
| 模块名称 | 核心功能 |
| :--- | :--- |
| **快捷记录模块** | 一键录音、语音转文字（ASR）、时间戳记录、快速标签添加。 |
| **日志处理模块** | 日志自动/手动分段、时间段计算、文本编辑、记录补录。 |
| **数据回顾模块** | 日/周/月视图切换、历史日志查询、时间轴展示。 |
| **复盘总结模块** | 每日/周/月总结输入、时间数据可视化（饼图/柱状图）。 |
| **设置与同步模块** | 用户设置、数据云同步、数据导出。 |

## 3. 功能需求 (Functional Requirements)

### 3.1. FR-1：快捷语音记录功能

| ID | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **FR-1.1** | **一键录音**：提供醒目的主操作按钮（如悬浮按钮），用户点击后立即开始录音，再次点击停止。 | 录音开始和停止的响应时间需小于 100ms。 |
| **FR-1.2** | **自动时间戳**：录音开始时，系统自动记录精确的**开始时间 (StartTime)**。 | 记录的时间戳精度至秒。 |
| **FR-1.3** | **语音转文字**：录音停止后，系统自动调用 ASR 服务将音频转为文本内容。 | ASR 转化成功率（普通话环境）不低于 95%。 |
| **FR-1.4** | **快速标签添加**：在 ASR 结果下方，提供一组可配置的**预设标签**（如：工作、学习、家庭、通勤、休闲），供用户一键标记。 | 用户可设置和管理不少于 10 个自定义标签。 |

### 3.2. FR-2：日志生成与编辑功能

| ID | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **FR-2.1** | **时间段自动划分**：基于两次记录的 **StartTime**，自动计算并生成上一条记录的 **EndTime**（即本次记录的 StartTime）。 | 自动生成的时间段格式为 `HH:MM - HH:MM`。 |
| **FR-2.2** | **日志文本编辑**：用户可以随时修改 ASR 转换后的文本内容。 | 文本修改后实时保存。 |
| **FR-2.3** | **时间段手动调整**：用户可以手动调整任何一条记录的 StartTime 和 EndTime。 | 调整 EndTime 时，如果与下一条记录的 StartTime 冲突，系统应提示并自动同步下一条记录的 StartTime。 |
| **FR-2.4** | **空白时间段补录**：当两条记录之间的时间间隔超过 3 小时（可配置），系统在日视图中高亮提示，引导用户进行“补录”操作。 | “补录”操作允许用户手动输入 StartTime 和 EndTime 以及对应的活动内容。 |
| **FR-2.5** | **情景模版填充**：允许用户创建常用模版（如：通勤路上，固定内容），点击后自动填充文本内容，减少语音录入。 | 模版内容和名称用户可自定义。 |

### 3.3. FR-3：回顾与总结功能

| ID | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **FR-3.1** | **日视图 (Daily View)**：按时间轴顺序展示一天的所有日志记录，包括时间段、活动文本、标签和地点。 | 视图默认展示今日内容，支持日期切换。 |
| **FR-3.2** | **周视图 (Weekly View)**：展示本周 7 天的每日总结，并提供一个**周总结输入框**。 | 周视图支持左右滑动切换周。 |
| **FR-3.3** | **月视图 (Monthly View)**：展示本月的每周总结，并提供一个**月总结输入框**。 | 月视图以日历形式展示，点击日期可跳转至日视图。 |
| **FR-3.4** | **每日总结**：在日视图下方，提供一个专门的输入框，供用户记录当日的核心产出和复盘心得。 | 总结内容与当日日志数据独立存储。 |
| **FR-3.5** | **时间可视化**：在周/月总结页面，根据用户记录的**标签**，生成**时间占比饼图**或**柱状图**。 | 图表应清晰显示“工作时间占比”、“学习时间占比”等关键数据。 |

### 3.4. FR-4：设置与数据管理

| ID | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **FR-4.1** | **数据云同步**：支持用户数据（日志、总结、标签设置）的云端同步和备份。 | 用户在更换设备后，登录小程序可恢复所有数据。 |
| **FR-4.2** | **数据导出**：支持用户按指定日期范围或标签筛选，将日志数据导出为 TXT 或 CSV 文件。 | 导出的文件内容格式清晰，包含时间段、活动内容和标签。 |
| **FR-4.3** | **关键提醒设置**：用户可设置每日固定时间点的提醒，点击提醒可直接跳转至小程序进行记录或复盘。 | 提醒功能需兼容小程序的服务通知或本地通知机制。 |

## 4. 非功能性需求 (Non-Functional Requirements)

### 4.1. 性能要求 (Performance Requirements)
* **PN-1：启动速度**：小程序冷启动时间应在 **3 秒内**完成。
* **PN-2：ASR 响应**：录音停止后，语音转文字结果应在 **3 秒内**显示。
* **PN-3：数据加载**：日视图、周视图、月视图的加载时间应在 **2 秒内**完成。

### 4.2. 可用性要求 (Usability Requirements)
* **US-1：界面简洁**：核心界面（录音、日视图）设计需极度简洁，信息展示清晰。
* **US-2：单手操作**：所有核心操作（录音、停止、标签选择）需适应单手操作习惯。
* **US-3：新手引导**：首次使用时，需提供简短的新手引导，教授一键录音和时间段划分逻辑。

### 4.3. 平台与兼容性 (Platform Requirements)
* **PL-1：平台兼容**：必须兼容 **微信小程序** 环境，并适配主流 iOS 和 Android 手机型号。
* **PL-2：权限请求**：录音权限的请求需在用户首次进行相应操作时进行，并给出清晰的请求目的说明。

### 4.4. 安全性要求 (Security Requirements)
* **SE-1：数据加密**：用户在云端存储的日志数据需进行加密处理。
* **SE-2：隐私保护**：严格遵守小程序平台的用户数据与隐私保护规范，未经用户授权，不得共享任何记录内容。

---

## 5. 约束与限制 (Constraints and Limitations)

### 5.1. 平台限制
* **小程序包体大小**：遵循小程序平台的包体限制（如微信小程序主包 2MB 限制）。
* **后台运行**：小程序在后台运行时，录音和定位功能可能会受到限制，需告知用户保持前台运行。

### 5.2. 技术选型（待定）
* **ASR 服务**：需选择成熟稳定的第三方 ASR API（如腾讯云、阿里云等），以保证转换效率和准确率。

### 5.3. 数据存储
* **云服务**：建议使用小程序平台提供的云开发/云存储服务，以简化运维和数据同步。

---

## 6. 用户界面要求 (User Interface Requirements)

### 6.1. UI 风格
* 整体采用**深色模式或高对比度**设计，突出信息重点，减少视觉干扰。

### 6.2. 核心界面设计
1.  **首页/录音页**：
    * 占据屏幕 80% 区域的**巨型录音按钮**。
    * 顶部显示当前时间或当日进度。
2.  **日志处理页**：
    * 清晰展示 ASR 文本，提供编辑入口。
    * 底部是**快速标签**选择区域。
3.  **日视图页**：
    * 采用简洁的**垂直时间轴**设计，记录项之间的时间间隔通过轴线长度直观体现。
    * 高亮显示 FR-2.4 所述的“空白时间段”。
